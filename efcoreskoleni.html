Základy

Co je ORM?

ORM (Object-Relational Mapping) – technika, která automaticky mapuje objekty v kódu na tabulky relační databáze a zpět. Místo psaní SQL pracujete s běžnými C# objekty.

Objektový svět (C#)

  • třídy, objekty a reference
  • kolekce (List<Post>), dědičnost
  • chování + data pohromadě

Relační svět (databáze)

  • tabulky, řádky a sloupce
  • primární a cizí klíče, JOINy
  • dotazování přes SQL
Tyto dva světy si vzájemně nerozumí – tzv. object-relational impedance mismatch. Bez ORM ho překlenujete ručně: SqlConnection, SqlCommand, DataReader a mapování řádek po řádku…
s ORM stačí
var nejlepsi = await db.Blogs.Where(b => b.Rating >= 4).ToListAsync();

Základy

Základní stavební kameny

  • Entita – obyčejná C# třída (POCO)jedna instance = jeden řádek v tabulce
  • DbContext – brána do databázespojení, konfigurace, change tracking; představuje jednu „jednotku práce“ (unit of work)
  • DbSet<T> – kolekce entit ≈ tabulkavstupní bod pro LINQ dotazy
  • Change tracker – hlídá změny načtených objektů
  • SaveChanges() – promítne všechny změny do databáze v jedné transakci
Model.cs
public class Blog
{
    public int Id { get; set; }      // PK dle konvence
    public required string Title { get; set; }
    public int Rating { get; set; }
    public List<Post> Posts { get; } = [];  // vztah 1:N
}

public class BlogContext : DbContext
{
    public DbSet<Blog> Blogs => Set<Blog>();
    public DbSet<Post> Posts => Set<Post>();
}
Základy

Jak vypadá práce s EF Core

Program.cs
using var db = new BlogContext();

// 1) DOTAZ – LINQ se přeloží na SQL
var top = await db.Blogs
    .Where(b => b.Rating >= 4)           // WHERE Rating >= 4
    .OrderByDescending(b => b.Rating)    // ORDER BY Rating DESC
    .Take(10)                            // TOP(10)
    .ToListAsync();

// 2) VLOŽENÍ nového záznamu
db.Blogs.Add(new Blog { Title = "Novinky v .NET 10" });

// 3) ÚPRAVA – change tracker si změny hlídá sám
top[0].Rating = 5;

// 4) ULOŽENÍ – INSERT + UPDATE v jedné transakci
await db.SaveChangesAsync();
  • Dotaz se vykoná až u ToListAsync(), First(), Count()…do té doby jen skládáte výraz (deferred execution)
  • Výsledné řádky se materializují na objekty Blog
  • Načtené entity jsou sledované – stačí změnit vlastnost
  • SaveChanges() pošle všechny změny najednou
Žádné ruční SQL, žádné mapování – jedna jednotka práce: otevřít context → pracovat → uložit → zavřít.
Přístupy

Dva směry: Code First vs. Database First

Přístupy

Code First – model píšete v C#

  • Kód je zdroj pravdy – databáze vzniká z entit a konfigurace
  • Model popisují konvence (Id = primární klíč), datové anotace a Fluent API v OnModelCreating
  • Schéma se verzuje v Gitu spolu s kódem – změny DB projdou code review
  • Ideální pro nové projekty (greenfield)
Pořadí síly: Fluent API přebíjí anotace, anotace přebíjí konvence.
Blog.cs + BlogContext.cs
public class Blog
{
    public int Id { get; set; }            // PK podle konvence
    [MaxLength(200)]                       // datová anotace
    public required string Title { get; set; }
    public List<Post> Posts { get; } = [];
}

public class BlogContext : DbContext
{
    public DbSet<Blog> Blogs => Set<Blog>();

    protected override void OnModelCreating(ModelBuilder mb)
    {
        mb.Entity<Blog>()                  // Fluent API
          .HasIndex(b => b.Title).IsUnique();
    }
}

Code First – migrace databáze

terminál
# jednorázově: CLI nástroj + balíček Design
dotnet tool install --global dotnet-ef
dotnet add package Microsoft.EntityFrameworkCore.Design

# 1) první migrace z aktuálního modelu
dotnet ef migrations add InitialCreate

# 2) aplikace migrací do databáze
dotnet ef database update

# … změním model … a znovu:
dotnet ef migrations add PridanoHodnoceni
dotnet ef database update

# SQL skript pro nasazení (produkce)
dotnet ef migrations script --idempotent -o deploy.sql
  • Migrace = C# třída s metodami Up() a Down()
  • EF si drží snapshot modelu – pozná, co se od minula změnilo
  • Aplikované migrace eviduje tabulka __EFMigrationsHistorydatabase update doplní jen ty chybějící
  • Migrace patří do Gitu – historie schématu jako součást kódu
Každá změna modelu → nová migrace. Migrace se skládají za sebe jako commity.

EF Core v praxi

Konzole – context si vytváříte ručně

AppDbContext.cs – varianta A: OnConfiguring
public class AppDbContext : DbContext
{
    public DbSet<Blog> Blogs => Set<Blog>();

    protected override void OnConfiguring(
        DbContextOptionsBuilder options)
        => options.UseSqlServer(
            "Server=.;Database=Blogy;…");
            // ⚠ radši číst z konfigurace
}

// Program.cs – vytvoření i úklid je na vás
using var db = new AppDbContext();
var blogy = await db.Blogs.ToListAsync();
varianta B: options zvenku
var options =
    new DbContextOptionsBuilder<AppDbContext>()
        .UseSqlServer(connString)
        .Options;

using var db = new AppDbContext(options);
  • Vy vytváříte, vy uklízíte – vždy using, ať se context korektně disposne
  • Varianta B je testovatelnější – konfigurace přichází zvenku (např. SQLite pro testy)
  • Stejný princip pro skripty, workery i desktop bez DI
EF Core v praxi

WPF a desktop – na co myslet

  • I v desktopu platí: context = krátká jednotka prácene jeden globální context po celou dobu běhu aplikace
  • Typicky nový context na operaci – načtení seznamu, uložení formuláře…
  • Context není thread-safe – pozor na práci na pozadí a UI vlákno
  • S Generic Hostem ve WPF: AddDbContextFactory a IDbContextFactory<T>továrna vyrábí čerstvé contexty; Dispose je pak na vás
Dlouhožijící context v desktopu = rostoucí paměť (change tracker drží všechna načtená data) a zastaralé údaje.
App.xaml.cs + ViewModel
// registrace v Generic Hostu (App.xaml.cs)
services.AddDbContextFactory<AppDbContext>(o =>
    o.UseSqlServer(
        config.GetConnectionString("Default")));

// ViewModel – na každou operaci čerstvý context
public class BlogsViewModel(
    IDbContextFactory<AppDbContext> factory)
{
    public async Task LoadAsync()
    {
        using var db = factory.CreateDbContext();
        Blogs = await db.Blogs
            .AsNoTracking()
            .ToListAsync();
    }
}

EF Core v praxi

Jak DI řídí životnost – scoped

  • AddDbContext registruje context jako scoped – 1 request = 1 instance
  • Žádné using – Dispose proběhne automaticky s koncem scope
  • AddDbContextPool – recyklace instancí, méně alokací při zátěži
  • Pozor: context nikdy neschovávat do singletonu; Blazor Server → IDbContextFactory (scope tam žije celý okruh)

Tipy & triky

Pozor na životnost a vlákna

  • Context je levný na vytvoření – vytvářejte klidně často, ale nedržte dlouho
  • Není thread-safe – jedna instance = jedna operace v daný okamžik
  • Vždy await – žádný „fire-and-forget“ SaveChangesAsync()
  • Paralelní práce → více contextů (factory / samostatné scopy)
tuhle chybu uvidíte…
InvalidOperationException: A second operation was
started on this context instance before a previous
operation completed…
typický scénář
// ❌ dvě operace zároveň na jednom contextu
var t1 = db.Blogs.ToListAsync();
var t2 = db.Posts.ToListAsync();
await Task.WhenAll(t1, t2);        // 💥 výjimka

// ✅ postupně na jednom contextu…
var blogs = await db.Blogs.ToListAsync();
var posts = await db.Posts.ToListAsync();

// ✅ …nebo opravdu paralelně, ale každý
//    dotaz s vlastním contextem z factory
using var dbA = factory.CreateDbContext();
using var dbB = factory.CreateDbContext();
Tipy & triky

N+1 problém – nejčastější zabiják výkonu

od nejhoršího k nejlepšímu
// ❌ N+1: 1 dotaz na blogy + 1 pro KAŽDÝ blog
foreach (var blog in await db.Blogs.ToListAsync())
    Console.WriteLine(blog.Posts.Count);
    // lazy loading → další SELECT v každé iteraci

// ✅ eager loading – souvislosti jedním dotazem
var blogs = await db.Blogs
    .Include(b => b.Posts)
    .ToListAsync();

// ✅✅ projekce – jen data, která potřebujete
var prehled = await db.Blogs
    .Select(b => new { b.Title,
                       Pocet = b.Posts.Count })
    .ToListAsync();
  • Lazy loading je pohodlný, ale zákeřný – smyčka nad entitami = smršť dotazů
  • Include() načte související data jedním dotazem (JOIN); jde i filtrovat
  • Projekce přes Select bývá nejrychlejší – méně sloupců, žádné sledování
  • AsNoTracking() pro čtení – citelně rychlejší a úspornější
Zapněte si ve vývoji logování SQL (LogTo) – uvidíte černé na bílém, kolik dotazů EF opravdu posílá.
Tipy & triky

Výkonové desatero (zkrácené)

AsNoTracking()

Čtení bez change trackeru – rychlejší a méně paměti. Default pro read-only dotazy.

Projekce (Select)

Tahejte jen potřebné sloupce, ne celé entity se vším všudy.

Stránkování

Skip / Take vždy s OrderBy; velká data → keyset pagination.

AsSplitQuery()

Více Include = kartézská exploze řádků. Split query ji rozdělí na víc dotazů.

ExecuteUpdate / Delete

Hromadné změny jedním SQL příkazem – bez načítání entit do paměti.

Async všude

ToListAsync, SaveChangesAsync… neblokujte vlákna při I/O.

hromadný update (EF 7+, v EF 10 i pro JSON sloupce)
await db.Blogs.Where(b => b.Rating < 2)
    .ExecuteUpdateAsync(s => s.SetProperty(b => b.Archived, true));
// → jeden UPDATE Blogs SET Archived = 1 WHERE Rating < 2
Tipy & triky

Migrace v produkci & tajemství

  • Aplikovanou migraci nikdy zpětně neupravovat – vždy přidat novou
  • Vygenerovanou migraci si přečtěte (hlavně DROP…) než ji pustíte dál
  • Nasazení: SQL skript (--idempotent) nebo migration bundleDatabase.Migrate() při startu jen s rozmyslem – více instancí = souběh
  • EnsureCreated() jen pro prototypy a testy – nemíchat s migracemi
  • Před migrací v produkci: záloha. Vždy.

Connection stringy & tajemství

  • nikdy do Gitu – vývoj: user secrets
  • produkce: proměnné prostředí / Key Vault
  • EnableSensitiveDataLogging jen ve vývoji
terminál
dotnet user-secrets init
dotnet user-secrets set "ConnectionStrings:Default" "Server=…"

# samospustitelný balíček migrací pro CI/CD
dotnet ef migrations bundle --self-contained

JSON sloupce

Value converters

Raw SQL

Pokročilé

Novinky v EF Core 10 (výběr)

  • LeftJoin / RightJoin – přímo jako LINQ operátory
  • Pojmenované query filtry – více filtrů na entitu, cílené vypínání
  • ExecuteUpdateAsync s běžnou lambdou – podmíněné SetProperty
  • Complex types – JSON mapování, podpora struct
  • Vektorové vyhledávání na Azure SQL – AI / RAG scénáře
  • Cosmos DB: fulltext + hybridní vyhledávání (RRF)
  • Chytřejší parametrizace kolekcí – stabilnější plan cache
  • Bezpečnost: redakce hodnot v lozích, analyzátor SQL injection
ochutnávka
// LeftJoin jako plnohodnotný operátor
var q = db.Students.LeftJoin(db.Departments,
    s => s.DepartmentId, d => d.Id,
    (s, d) => new { s.Name, Dept = d.Name ?? "–" });

// pojmenované query filtry…
mb.Entity<Blog>()
  .HasQueryFilter("SoftDelete", b => !b.IsDeleted)
  .HasQueryFilter("Tenant",
      b => b.TenantId == tenantId);

// …a jejich cílené vypnutí v dotazu
var vse = await db.Blogs
    .IgnoreQueryFilters(["SoftDelete"])
    .ToListAsync();
Závěr

Co si odnést

  • ORM mapuje objekty ⇄ tabulky – EF Core je standardní ORM pro .NET; vy píšete LINQ, SQL vzniká pod kapotou
  • Code First = model v C# + migrace  ·  Database First = scaffolding existující databáze (klidně jen jednou na startu)
  • Konzole / WPF: context si vytváříte a uklízíte sami  ·  ASP.NET Core: AddDbContext + DI, scoped na request
  • DbContext: krátká životnost, není thread-safe, vždy await
  • Výkon: AsNoTracking, Include / projekce místo N+1, stránkování, ExecuteUpdate
  • EF Core 10 + .NET 10 = LTS do listopadu 2028 – solidní základ na další roky
