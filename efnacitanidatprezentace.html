Tři nástroje na tři problémy

„Načetl jsem blogy, ale blog.Posts je prázdné…“

Navigační vlastnosti se samy nenačítají – a lazy loading ve smyčce znamená N+1 dotazů.

.Include(…)

„Data jen zobrazuji – proč si je EF všechny pamatuje?“

Change tracker sleduje každou načtenou entitu. U read-only dotazů je to čistá režie navíc.

Include

Include – eager loading souvisejících dat

eager loading
// ❌ navigace se samy nenačítají
var ble = await db.Blogs.ToListAsync();
var pocet = ble[0].Posts.Count;      // 0 – nikdo je nenačetl
                                     // (s lazy loadingem: N+1!)

// ✅ Include – vše v jednom dotazu (JOIN)
var blogy = await db.Blogs
    .Include(b => b.Posts)           // kolekce 1:N
    .Include(b => b.Owner)           // reference N:1
    .ToListAsync();

// ✅ více úrovní – ThenInclude
var strom = await db.Blogs
    .Include(b => b.Posts)
    .ThenInclude(p => p.Author)
    .ToListAsync();
  • Motivace: bez Include jsou navigace prázdné – nebo je lazy loading dočítá po jedné (N+1)
  • Include řekne EF, co načíst rovnou s entitou
  • ThenInclude pokračuje o úroveň hlouběji
  • Víc Include = víc vztahů najednouale pozor na více kolekcí – viz AsSplitQuery
Include vrací celé entity. Když potřebujete jen pár sloupců, bývá lepší projekce přes Select.
Include

Filtrovaný Include a AutoInclude

jen část kolekce
// filtrovaný Include – ne celá kolekce
var blogy = await db.Blogs
    .Include(b => b.Posts
        .Where(p => !p.IsDraft)
        .OrderByDescending(p => p.PublishedOn)
        .Take(5))                    // top 5 postů na blog
    .ToListAsync();

// AutoInclude – navigace se načítá vždy (model)
modelBuilder.Entity<Theme>()
    .Navigation(t => t.ColorScheme)
    .AutoInclude();

// …a vypnutí pro konkrétní dotaz
var cista = await db.Themes
    .IgnoreAutoIncludes()
    .ToListAsync();
  • Uvnitř Include jde: Where, OrderBy(Descending), ThenBy, Skip, Take
  • Typické použití: „posledních 5 komentářů ke každému článku“
  • Pozor s trackingem: už sledované entity se do kolekce doplní taky (fixup) – přesný výsledek zaručí AsNoTracking
  • AutoInclude šetří psaní u navigací, které chcete opravdu vždy
No-tracking

AsNoTracking – čtení bez sledování změn

read-only dotazy
// ✅ jen zobrazuji → bez change trackeru
var blogy = await db.Blogs
    .AsNoTracking()
    .ToListAsync();

// výchozí chování pro celý context…
optionsBuilder.UseQueryTrackingBehavior(
    QueryTrackingBehavior.NoTracking);

// …a zpět sledování tam, kde edituji
var blog = await db.Blogs
    .AsTracking()
    .FirstAsync(b => b.Id == id);

blog.Rating = 5;
await db.SaveChangesAsync();   // vidí jen sledované!
  • Motivace: change tracker si drží snapshot každé entity a porovnává změny – čas i paměť navíc, které u čtení nevyužijete
  • Bez sledování je dotaz citelně rychlejší a úspornější
  • Ale: změny takových entit SaveChanges() neuloží
  • Duplicitní entita = nová instance → případně AsNoTrackingWithIdentityResolution()
  • Projekce (Select bez celé entity) se nesledují nikdy
Pravidlo: čtu → AsNoTracking, edituji → tracking.
Split query

Problém: kartézská exploze

dvě kolekce vedle sebe
var blogy = await db.Blogs
    .Include(b => b.Posts)           // 10 postů
    .Include(b => b.Contributors)    // 10 autorů
    .ToListAsync();
vygenerované SQL (jeden dotaz)
SELECT …
FROM Blogs b
LEFT JOIN Posts p        ON b.Id = p.BlogId
LEFT JOIN Contributors c ON b.Id = c.BlogId
-- křížový součin: každý post × každý autor

1 blog · 10 postů · 10 autorů

10 × 10 = 100 řádků

a sloupce blogu se opakují v každém z nich

  • JOIN dvou kolekcí na stejné úrovni vrací jejich křížový součin
  • Duplicity = zbytečná data po síti i v paměti
  • EF Core vás sám varuje (MultipleCollectionIncludeWarning), když načítáte víc kolekcí jedním dotazem
Split query

Řešení: AsSplitQuery – a jeho kompromisy

rozdělení na více dotazů
// ✅ místo jednoho JOINu více dotazů
var blogy = await db.Blogs
    .Include(b => b.Posts)
    .Include(b => b.Contributors)
    .AsSplitQuery()
    .ToListAsync();
// → 3 SQL dotazy: Blogs, Posts, Contributors

// globální výchozí chování…
optionsBuilder.UseSqlServer(conn, o =>
    o.UseQuerySplittingBehavior(
        QuerySplittingBehavior.SplitQuery));

// …a výjimka zpět na jeden dotaz
var mensi = await db.Blogs
    .Include(b => b.Owner)
    .AsSingleQuery()
    .ToListAsync();
  • Každá kolekce svým dotazem → žádná exploze ani duplicity
  • Cena: více round-tripů do databáze (latence, hlavně v cloudu)
  • Konzistence: mezi dotazy může někdo data změnit – řeší transakce / snapshot izolace
  • Reference (N:1) zůstávají JOINem i ve split režimu
  • EF Core 10 opravilo řazení a Skip/Take u split queries – dřív hrozily nekonzistentní výsledky
Doporučení: default nechte single, AsSplitQuery zapínejte u dotazů s více kolekcemi – EF vás na ně varováním upozorní sám.
