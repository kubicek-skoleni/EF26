Datové anotace v EF Core

Referenční přehled atributů, kterými se Entity Frameworku předávají informace o modelu přímo v kódu entit. Roztříděno podle účelu; u každého atributu je uveden jmenný prostor a případně verze EF Core, od které je k dispozici.

Klíče a identita

Primární klíče, generování hodnot a optimistická souběžnost.

[Key]
DataAnnotations
Označí vlastnost jako primární klíč. Bez atributu platí konvence: vlastnost Id nebo <NázevTřídy>Id.[Key] public Guid CisloObjednavky { get; set; }
[PrimaryKey(nameof(A), nameof(B))]
EF CoreEF 7+
Složený primární klíč – atribut se dává na třídu. Pozor: více [Key] atributů v EF Core nefunguje; složený klíč jde jen takto, nebo přes Fluent API HasKey.[PrimaryKey(nameof(StudentId), nameof(PredmetId))] public class Zapis { … }
[DatabaseGenerated(…)]
…Schema
Určuje, jak databáze generuje hodnotu: Identity (při INSERT), Computed (při INSERT i UPDATE), None (nikdy – vypne identity).[DatabaseGenerated(DatabaseGeneratedOption.None)] public int Id { get; set; } // hodnotu dodává aplikace
[Timestamp]
DataAnnotations
Sloupec rowversion – automatický token optimistické souběžnosti. Při UPDATE se hodnota přidá do WHERE; konflikt vyhodí DbUpdateConcurrencyException.[Timestamp] public byte[] Verze { get; set; }
[ConcurrencyCheck]
DataAnnotations
Zahrne konkrétní vlastnost do kontroly souběžnosti (WHERE při UPDATE/DELETE) – bez rowversion sloupce. Hodnotu ale musí měnit aplikace sama.

Tabulky, sloupce a mapování typů

Kam a jak se třída či vlastnost mapuje – a co se nemapuje vůbec.

[Table("blogy", Schema = "web")]
…Schema
Mapuje třídu na konkrétní tabulku (a schéma). Bez atributu se použije název DbSet vlastnosti.
[Column("blog_id", TypeName = "varchar(200)", Order = 0)]
…Schema
Nastaví název sloupce, jeho databázový typ a pořadí při vytváření tabulky. Vše je volitelné – lze použít jen část.
[Comment("Firemní blogy")]
EF Core
Přidá komentář k tabulce nebo sloupci přímo v databázi (dostane se do migrací). Skvělé pro dokumentaci schématu.
[NotMapped]
…Schema
Vyloučí vlastnost nebo celou třídu z mapování – existuje jen v paměti, v databázi nevznikne.[NotMapped] public string CeleJmeno => $"{Jmeno} {Prijmeni}";
[Keyless]
EF CoreEF 5+
Entita bez primárního klíče – typicky pro databázové pohledy (views) nebo výsledky raw SQL dotazů. Nelze sledovat změny ani ukládat.
[Owned]
EF Core
Vlastněný typ (owned entity) – nemá vlastní identitu, žije uvnitř nadřazené entity. Pro nové projekty dnes zvažte spíše complex types (níže).
[ComplexType]
…SchemaEF 8+
Komplexní typ bez identity s hodnotovou sémantikou (např. Adresa). V EF Core 10 podporuje i struct, volitelnost (nullable) a mapování do JSON sloupce (ToJson() přes Fluent API).[ComplexType] public record struct Adresa(string Ulice, string Mesto, string Psc);
[EntityTypeConfiguration(typeof(BlogConfig))]
EF CoreEF 6+
Odkáže třídu na její konfigurační třídu IEntityTypeConfiguration<T> – Fluent API konfigurace se najde sama, bez registrace v OnModelCreating.

Povinnost, délky a přesnost

Atributy tvarující sloupce – NULL/NOT NULL, délky textů, přesnost čísel.

[Required]
DataAnnotations
Sloupec bude NOT NULL. Pozor: se zapnutými nullable reference types je non-nullable typ (string) povinný už konvencí – [Required] pak dává smysl hlavně pro string? nebo při vypnutých NRT.
[MaxLength(200)]
DataAnnotations
Maximální délka – z nvarchar(max) udělá nvarchar(200). Funguje pro string i pole bajtů. Preferovaný atribut pro EF.
[StringLength(200, MinimumLength = 3)]
DataAnnotations
Pro schéma totéž co MaxLength (jen pro string), navíc umí MinimumLength – ta ale slouží jen validaci, schéma nemění.
[MinLength(3)]
DataAnnotations
Jen validace – na databázové schéma nemá vliv (minimální délku sloupce SQL neumí). Uvádíme pro úplnost.
[Unicode(false)]
EF CoreEF 6+
Vypne Unicode – na SQL Serveru varchar místo nvarchar (poloviční místo). Hodí se pro ASCII data: ISBN, kódy, URL.
[Precision(18, 2)]
EF CoreEF 6+
Přesnost a měřítko – decimal(18,2). S jedním parametrem funguje i pro datum/čas: [Precision(3)] → datetime2(3).

Vztahy (relace)

Propojení navigačních vlastností, cizích klíčů a chování při mazání.

[ForeignKey(nameof(Blog))]
…Schema
Spáruje cizí klíč s navigační vlastností, když neodpovídají konvenci (BlogId). Lze umístit na FK vlastnost nebo na navigaci.[ForeignKey(nameof(Blog))] public int BlogKlic { get; set; } public Blog Blog { get; set; } = null!;
[InverseProperty(nameof(Post.Autor))]
…Schema
Spáruje navigaci s její protistranou – nutné, když mezi dvěma typy existuje více vztahů (např. autor a recenzent příspěvků).
[DeleteBehavior(DeleteBehavior.Restrict)]
EF CoreEF 7+
Nastaví chování při smazání principala: Cascade, Restrict, SetNull, ClientSetNull, NoAction… Dává se na navigaci na straně závislé entity. Konvence: povinný vztah = Cascade.
[Required] (na navigaci)
DataAnnotations
Na navigační vlastnosti označí vztah jako povinný – cizí klíč bude NOT NULL.

Indexy – [Index] a Fluent API

Aktuální syntaxe je stejná v .NET 9 (EF Core 9) i .NET 10 (EF Core 10). Atribut [Index] se dává na třídu a pochází z Microsoft.EntityFrameworkCore.

datové anotace – [Index] na třídě
using Microsoft.EntityFrameworkCore;   // ← odsud!

// jednoduchý index
[Index(nameof(Url))]

// unikátní index
[Index(nameof(Url), IsUnique = true)]

// složený index (na pořadí záleží!)
[Index(nameof(Prijmeni), nameof(Jmeno))]

// vlastní název
[Index(nameof(Url), Name = "IX_Blog_Url")]

// řazení sloupců (EF 7+)
[Index(nameof(Url), nameof(Rating),
       AllDescending = true)]
[Index(nameof(Url), nameof(Rating),
       IsDescending = new[] { false, true })]
public class Blog { … }
Fluent API – plná výbava (OnModelCreating)
modelBuilder.Entity<Blog>()
    .HasIndex(b => new { b.Url, b.Rating })
    .IsUnique()
    .IsDescending(false, true)
    .HasDatabaseName("IX_Blog_Url_Rating");

// filtrovaný index – JEN Fluent API
modelBuilder.Entity<Blog>()
    .HasIndex(b => b.Url)
    .HasFilter("[Url] IS NOT NULL");

// included sloupce – JEN Fluent API
modelBuilder.Entity<Post>()
    .HasIndex(p => p.Url)
    .IncludeProperties(p => new { p.Title });

// více indexů nad stejnými sloupci → názvy
modelBuilder.Entity<Person>()
    .HasIndex(p => new { p.Prijmeni }, "IX_Asc");
Pozor na záměnu s EF6: starý Entity Framework 6 měl [Index] ze System.ComponentModel.DataAnnotations.Schema a dával se na vlastnost. V EF Core tento atribut neexistuje – používá se [Index] z Microsoft.EntityFrameworkCore na úrovni třídy.
Konvence: na sloupce cizích klíčů vytváří EF Core index automaticky. Výchozí název indexu je IX_<Tabulka>_<Sloupce>.
.NET 9 vs .NET 10: syntaxe indexů (i všech anotací výše) je v EF Core 9 a EF Core 10 totožná. Jen připomínka: .NET 9 je STS s koncem podpory v květnu 2026 – pro nové projekty volte .NET 10 (LTS do listopadu 2028).

Ostatní

Méně časté, ale užitečné.

EF CoreEF 5+
Určí backing field, do kterého má EF číst/zapisovat místo vlastnosti – hodí se pro zapouzdření (vlastnost jen pro čtení, logika v setteru…).
[DbFunction]
EF Core
Namapuje statickou C# metodu na databázovou funkci, aby šla volat v LINQ dotazech a přeložila se do SQL.

Pozor: atributy, které schéma NEmění

Tyto validační atributy z System.ComponentModel.DataAnnotations využívá ASP.NET Core (model binding, Blazor formuláře) – EF Core je při tvorbě schématu ignoruje. Na školení je dobré tento rozdíl zdůraznit.

[Range(1, 100)] [RegularExpression(…)] [EmailAddress] [Phone] [Url] [CreditCard] [Compare(…)] [DataType(…)] [Display(…)] [DisplayFormat(…)] [MinLength(…)]
Výjimky, které schéma mění, jsou popsané výše: [Required] (NOT NULL) a [MaxLength] / [StringLength] (délka sloupce).
