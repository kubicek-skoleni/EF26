Logování SQL dotazů

Jak vidět, co EF Core opravdu posílá na SQL Server
Školení Entity Framework CoreEF Core 9 / 102026
Motivace

Je to zapnuté od začátku?

Konzolová aplikace – NE

Bez konfigurace se neloguje vůbec nic – výstup prostě není kam poslat. Zapíná se jedním řádkem:

.LogTo(Console.WriteLine)

Žádný NuGet balíček navíc není potřeba.

ASP.NET Core – v praxi ANO

EF se automaticky napojí na ILogger. Výchozí šablona loguje úroveň Information do konzole, takže SQL vidíte hned při dotnet run.

Řídí se to ale nastavením v appsettings.json – když si ho někdo ztlumí, SQL zmizí.

Platí pro všechny poskytovatele včetně Microsoft.EntityFrameworkCore.SqlServer – loguje samotné EF, takže vidíte přesně to SQL (včetně parametrů), které odchází na SQL Server.
Konzole / WPF

LogTo – jednoduché logování

AppDbContext.cs
public class AppDbContext : DbContext
{
    public DbSet<Blog> Blogs => Set<Blog>();

    protected override void OnConfiguring(
        DbContextOptionsBuilder options)
        => options
            .UseSqlServer("Server=.;Database=Blogy;…")
            .LogTo(Console.WriteLine,
                   LogLevel.Information);   // ✅ SQL v konzoli
}
varianta: jen SQL příkazy, nic jiného
.LogTo(Console.WriteLine,
       [DbLoggerCategory.Database.Command.Name],
       LogLevel.Information)
  • LogTo je nejrychlejší cesta – bez balíčků, jen jeden řádek v OnConfiguring
  • LogLevel.Information = provedené SQL příkazybez něj i spousta Debug hlášek navíc
  • Filtr kategorií Database.Command = čistý výpis jen SQL
  • Jiné cíle: Debug.WriteLine (okno Output ve VS), zápis do souboru přes StreamWriter
Konzole / WPF

Co uvidíte – a kde jsou hodnoty parametrů

výstup v konzoli
info: Microsoft.EntityFrameworkCore.Database.Command[20101]
      Executed DbCommand (12ms)
      [Parameters=[@__minRating_0='?' (DbType = Int32)],
       CommandType='Text', CommandTimeout='30']
      SELECT [b].[Id], [b].[Title], [b].[Rating]
      FROM [Blogs] AS [b]
      WHERE [b].[Rating] >= @__minRating_0
zobrazit skutečné hodnoty parametrů
options
    .LogTo(Console.WriteLine, LogLevel.Information)
    .EnableSensitiveDataLogging()   // '?' → '4'
    .EnableDetailedErrors();
  • Hodnoty parametrů jsou default skryté – ve výpisu je místo nich '?'
  • EnableSensitiveDataLogging() je odkryjedo logů se dostanou osobní údaje – jen pro vývoj!
  • EnableDetailedErrors() – podrobnější chybové hlášky
  • Z výpisu vidíte i čas provedení (12ms) – první vodítko při ladění výkonu
V produkci citlivé logování nikdy – a běžné logování SQL jen s rozmyslem (objem logů, výkon).
ASP.NET Core

ILogger + appsettings.json

appsettings.Development.json
{
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "Microsoft.AspNetCore": "Warning",

      "Microsoft.EntityFrameworkCore.Database.Command":
        "Information"
    }
  }
}
  • AddDbContext napojí EF na ILogger automaticky – v C# kódu není potřeba nic
  • Výchozí šablona (Default = Information) → SQL v konzoli uvidíte rovnou
  • Jistota = explicitní kategorie Microsoft.EntityFrameworkCore.Database.Commandfunguje nezávisle na tom, jak je ztlumený zbytek
  • Produkce: ztlumte ji na "Warning"
  • Přes ILogger to jde i do Serilogu, souborů, Seq…
Bonus

ToQueryString() a shrnutí

SQL jednoho dotazu – bez logování a bez spuštění
var dotaz = db.Blogs
    .Where(b => b.Rating >= 4)
    .OrderByDescending(b => b.Rating);

Console.WriteLine(dotaz.ToQueryString());
výstup
SELECT [b].[Id], [b].[Title], [b].[Rating]
FROM [Blogs] AS [b]
WHERE [b].[Rating] >= @__minRating_0
ORDER BY [b].[Rating] DESC
